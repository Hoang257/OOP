## Описание проекта
Реализация пользовательского класса `Athlete` (спортсмен) с применением инкапсуляции, свойств, магических методов и бизнес-логики.

## Ответы на контрольные вопросы

### Вопрос 1. Что является сущностью?
1) Я выбрал класс Athlete, как сущность. Сущность - это обьект который мы моделируем в программе

### Вопрос 2. Какие у него атрибуты?
2) У класса есть следующие **закрытые атрибуты** (приватные поля):
- `_id` - уникальный идентификатор спортсмена;
- `_name` - имя;
- `_age` - возраст;
- `_weight` - вес (кг);
- `_height` - рост (см);
- `_status` - активен ли спортсмен (`True`/`False`);
- `_num_visiting` - количество проведённых тренировок;
- `_paid` - оплачена ли текущая тренировка (`True`/`False`).

### Вопрос 3. Какие инварианты?
3) Инварианты - это условия, которые объект должен соблюдать всегда:
- `_id >= 0` и целое число;
- `_name` - непустая строка;
- `_age >= 16` (минимальный возраст);
- `_weight > 0`;
- `_height > 0`;
- `_status` - булево значение;
- `_num_visiting >= 0`;
- `_paid` - булево значение.
Эти инварианты проверяются при создании объекта и при изменении свойств через сеттеры.

### Вопрос 4. Что значит “равенство”?
4) Равенство двух обьектов класса определяется по i, два спорсмена считаются одинаковым, если у них id совпадают, независимо от остальных атрибутов. Это было реальизовано в магическом методе __eq__.

### Вопрос 5. Есть ли состояние?
5) Да, в моем случае состояние изменяется через методы: pay(), not_pay() - оплачена тренировка или нет, active(), deactive() - активный сейчас или нет. Они напрямую будут влиять на метод train().

## Цель лабораторной работы — познакомиться с абстрактными базовыми классами `ABC`, освоить понятие интерфейса как контракта поведения, научиться задавать обязательные методы для классов и закрепить полиморфизм через единый интерфейс.
На основе уже существующей системы (ЛР1–ЛР3) ввел контракты поведения, которые обязывают классы реализовать определённые методы(Printable, Comparable)

Интерфейс Printable задаёт контракт для объектов, которые можно представить в виде строки и он требует обязательный метод "to_string()"

Интерфейс Comparable задаёт контракт для объектов, которые можно сравнивать между собой и он требует обязательный метод "compare_to(other)"
1   - если текущий объект больше другого
-1  - если текущий объект меньше другого
0   - если объекты равны

описание методов:

Метод get_printable()

Возвращает новую коллекцию только с объектами, которые реализуют интерфейс Printable.

Метод get_comparable()

Возвращает новую коллекцию только с объектами, которые реализуют интерфейс Comparable.

Метод print_all_by_interface()

Выводит объекты через интерфейс Printable, при этом не нужно проверять тип обьекта

Метод sort_by_comparable()

Сортирует объекты через интерфейс Comparable.
